Lesson 4｜Handoff / 分工 / 上下文交接

這一課是多 agent 能不能真的落地的核心：切工作不難，難的是交接後還能不失真。 Handoff 做不好，分工越多，品質越碎。

Step 1｜先把 handoff 的本質講懂

一句話版本： Handoff 不是單純把工作丟給下一個 agent，而是把「下一個人做對事所需要的最小充分資訊」交清楚。

為什麼 handoff 會成為瓶頸？

因為 agent 之間不像人類團隊那樣真的共享腦袋。上一個 agent 知道的事，下一個 agent 不會自動知道。你如果只是說「請接手」，等於什麼都沒交。下一位不是重做，就是亂補。

所以 handoff 的難點在於：怎麼在有限上下文裡，把必要資訊交得完整，但又不把垃圾一起帶過去。

一個好的 handoff 至少要有

  • 任務目標：下一位到底要完成什麼。
  • 目前狀態：已做了哪些、做到哪裡。
  • 重要結論：前一步已經確認的事，不要讓下一位重算。
  • 限制與風險：不能碰什麼、哪裡不確定。
  • 輸出格式：你希望下一位交回什麼樣子。

常見失敗 handoff 長這樣

  • 只有一句「請接續完成」。
  • 把整包原始資料丟過去，沒有摘要與優先順序。
  • 沒有說哪些結論已定、哪些還是假設。
  • 沒有交輸出標準，讓下一位猜規格。
  • 把噪音也一起帶走，害下游判斷污染。

Handoff 其實像交班，不是甩鍋

你可以把它想成醫院交班、課程交接、剪接案從研究交給腳本。真正好的交接不是「我做完了，你看著辦」，而是「我已經幫你把路清到這裡，以下這些是你接下來最該看的資訊」。

如果上一位沒整理，下游就會面對三種壞事：

  1. 重新做一遍，浪費成本。
  2. 理解錯方向，直接偏航。
  3. 看太多雜訊，反而把關鍵忽略。

實作上怎麼減少失真？

  • 把交接內容結構化，例如目標 / 完成狀態 / 關鍵證據 / 未解問題。
  • 明確區分「已確認」與「待驗證」，避免假設被誤當結論。
  • 盡量傳遞必要摘要，而不是無差別傳遞全部原文。
  • 如果下一步很依賴原始證據，可附精簡引用，而不是整包回灌。

一句話：handoff 不是資料搬運，是判斷後的交班。

最重要的成熟觀念

很多系統不是敗在模型，而是敗在交接。前面做得很漂亮，最後一 handoff 寫得像便利貼，整個品質就斷掉。所以設計多 agent 時，不要只設計角色，也要設計交接語言。

龐德的見解

一個團隊真正的成熟度，往往不是看誰最會做，而是看交接時能不能不靠默契也不失真。Agent 系統更是如此，因為它連默契都沒有。

所以 handoff 設計到位，分工才是放大器；handoff 爛，分工就變成品質粉碎機。

最後一句話

Handoff 不是把工作甩出去，而是把下一步成功所需的資訊交到位。
