Lesson 2｜上下文工程與流程的關係

這一課要把一個關鍵誤區拆掉：context engineering 不是 prompt 小技巧，而是整個流程怎麼切、怎麼接、怎麼回填資訊的設計能力。

Step 1｜先把主軸講穿

一句話版本： 流程 決定事情分成幾步、每步做什麼；Context Engineering 決定每一步到底拿到哪些資訊。兩者分不開，因為步驟一變，該看的上下文就跟著變。

為什麼不能把它當成 prompt 技巧？

如果你只把 context engineering 理解成「開場白怎麼寫得比較漂亮」，你會忽略真正難的部分：任務是多步驟的，模型不是只回答一次就結束。 它可能先判斷要不要查資料，再讀工具輸出，再整理成下一步動作。每往前走一步，需要的資訊都不一樣。

所以真正的工程問題不是「第一句怎麼問」，而是：在第 2 步、第 3 步、第 8 步，哪些資訊還該保留？哪些資訊該壓縮？哪些該轉成結構化摘要？

流程 在管什麼？

  • 先規劃，還是先查資料？
  • 一個 agent 做到底，還是切給不同角色？
  • 什麼時候需要工具呼叫？
  • 失敗後要重試、改路徑，還是交給人？

Context Engineering 在管什麼？

  • 此刻模型該看到任務目標、限制、歷史中的哪一段？
  • 工具輸出是原文回灌，還是先摘要？
  • 哪些規則要永遠保留，哪些只在某一步出現？
  • 不同步驟之間的交接內容如何避免失真？

用工作場景講，會更清楚

假設你要做一支影片企劃。流程 可能是：需求釐清 → 找資料 → 出大綱 → 改稿 → 產出。那 context engineering 就是在每一步替模型準備 briefing：

  1. 需求釐清：要給目標受眾、影片目的、限制條件。
  2. 找資料：要給查詢方向與可信來源標準，不需要把後面剪接需求一起塞進去。
  3. 出大綱：要保留前面的核心結論，但不必保留每一筆搜索原始噪音。
  4. 改稿：要把前版內容、修改要求、不能動的核心訊息一起交接。

你看，同一個任務，不同步驟需要不同上下文。這就是為什麼 流程 一設計，context engineering 就一定跟著來。

真正成熟的系統會做的事

  • 不是把所有聊天紀錄全丟回去，而是依步驟重組。
  • 不是每個工具輸出都原封不動回灌，而是先過濾與壓縮。
  • 不是讓模型一直自由發揮，而是替它設計好「現在你該做哪一類判斷」。

結論：流程 是骨架，Context Engineering 是送到每個關節的血液。骨架沒有血液，動不起來；只有血液沒有骨架，也只是一灘。

龐德的見解

很多人把 agent 系統做爛，不是模型不夠強，而是沒有把 流程 與 context 一起設計。步驟亂切，資訊亂塞，最後模型像在霧裡開車。

真正好的設計，不是讓模型永遠知道全部，而是讓模型在每一步都知道現在最該知道的那一部分。

最後一句話

流程 決定做事順序，Context Engineering 決定每一步看到的資訊；兩者一起設計，agent 才會穩。
