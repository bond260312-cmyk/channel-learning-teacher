Lesson 3｜單 Agent 與多 Agent 的差別

這一課不是要你迷信多 agent，而是要你知道：多不一定比較強，分工是否值得，取決於任務複雜度、交接成本與失真風險。

Step 1｜先把判斷標準講清楚

一句話版本： 單 agent 像一個能幹的總召，從頭做到底；多 agent 像一個團隊，把工作切給不同角色。差別不在「看起來專業」，而在切分之後到底有沒有比單做更穩、更快、更可控。

單 Agent 的優點與限制

  • 優點：上下文集中、交接少、實作簡單、比較容易除錯。
  • 適合：任務範圍明確、步驟不多、同一種思維能一路做到底的事情。
  • 限制：一個 agent 同時扛規劃、查詢、整理、判斷時，容易上下文變胖、角色混雜、注意力分散。

多 Agent 的優點與限制

  • 優點：可以角色專門化，例如 researcher、writer、reviewer 各做各的。
  • 適合：任務長、異質性高、需要不同判斷框架，或希望把責任拆清楚。
  • 限制：每多一個 agent，就多一次交接、多一份上下文整理、多一層延遲與失真風險。

所以多 agent 不是免費午餐。你得到專門化，也付出協調成本。

什麼情況用單 Agent 比較合理？

  1. 目標單一，例如把一段會議內容整理成摘要。
  2. 資料量還在可控範圍內，不需要複雜 handoff。
  3. 你重視速度與維護簡單度。
  4. 錯誤來源要容易追查，不想開太多角色線。

什麼情況多 Agent 會更有價值？

  1. 任務本身包含明顯不同職能，例如研究、規劃、審核。
  2. 單 agent 已開始上下文超載，做事品質不穩。
  3. 你希望把輸出分層驗證，而不是一次押寶。
  4. 你需要模組化替換，例如 reviewer 想換模型或規則。

實務上最重要的不是「多」，而是「切得對」

很多人一學 agent 就很興奮，恨不得拆成十幾個角色：企劃 agent、摘要 agent、語氣 agent、排版 agent、審核 agent……結果系統變得很熱鬧，卻沒有更準。原因很簡單：切太細，交接成本反而大過收益。

好的切法通常符合三個條件：

  • 每個 agent 有明確職責，不是名字不同但其實都在做差不多的事。
  • 交接介面清楚，知道上一位要交什麼，下一位才接得住。
  • 切完後真的提升品質、穩定性或可維護性，而不是只是看起來高級。

一句白話比喻

單 agent 像一個很強的 PM＋執行者，一個人跑完整專案；多 agent 像把專案拆給研究員、企劃、審稿、執行。如果專案很小，一個人做更快；如果專案很大，分工才有意義。 但分工前提是你真的會管理，不然只是把混亂拆成好幾份混亂。

龐德的見解

單 agent 與多 agent 的差別，本質上不是架構美學，而是管理學。你不是在拼角色數量，而是在拼：這樣切，系統有沒有更穩？

真正成熟的人，不會逢人就上多 agent；會先問一句：這件事真的值得拆嗎？ 如果不值得，就別把簡單問題做成豪華車禍現場。

最後一句話

單 agent 贏在集中與簡單，多 agent 贏在分工與模組化；關鍵不是誰高級，而是誰更適合這個任務。
